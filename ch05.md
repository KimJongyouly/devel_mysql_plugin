# 제5장: 커스텀 스토리지 엔진 — Parquet 파일 읽기

---

## 5.1 스토리지 엔진 아키텍처

### MySQL의 스토리지 계층

MySQL은 SQL 파서, 옵티마이저, 스토리지 계층이 명확하게 분리된 구조입니다.

```
클라이언트 SQL
    ↓
파서 → 옵티마이저 → 실행 엔진
                        ↓
              ┌──────────────────────┐
              │  스토리지 엔진 추상층  │  ← handler 추상 클래스
              └──────────────────────┘
              ↙         ↓         ↘
           InnoDB    MyISAM    READ_PARQUET
```

쿼리 실행 엔진은 `handler` 추상 클래스의 인터페이스만 사용합니다. 어떤 스토리지 엔진인지는 신경 쓰지 않습니다. 따라서 `handler`를 상속받아 필요한 메서드를 구현하면 새로운 스토리지 엔진을 만들 수 있습니다.

이 장의 `READ_PARQUET`는 `.parquet` 파일을 MySQL 테이블처럼 쿼리할 수 있게 해주는 **읽기 전용** 스토리지 엔진입니다.

```sql
CREATE TABLE sales (
    order_id INT,
    product VARCHAR(128),
    amount DOUBLE
) ENGINE=READ_PARQUET
  COMMENT='/data/warehouse/sales_2024.parquet';

SELECT product, SUM(amount) FROM sales GROUP BY product;
```

### Apache Parquet 포맷

Apache Parquet는 컬럼 지향(columnar) 바이너리 파일 포맷입니다. 데이터 웨어하우스, 빅데이터 파이프라인에서 널리 사용됩니다.

- **컬럼 지향**: 같은 컬럼의 값들이 연속 저장 → 집계 쿼리에 효율적
- **스키마 내장**: 파일 자체에 컬럼명, 타입 정보가 포함
- **압축**: Snappy, Gzip 등 다양한 압축 지원

이 엔진은 Apache Arrow C++ 라이브러리(`libarrow`, `libparquet`)로 Parquet 파일을 파싱합니다.

## 5.2 핵심 클래스 설계

### Parquet_share — 테이블 수준 공유 상태

```cpp
class Parquet_share : public Handler_share {
 public:
  THR_LOCK lock;                  // 테이블 레벨 락
  char     table_name[FN_REFLEN]; // 테이블 경로 (확장자 제외)

  Parquet_share()  { thr_lock_init(&lock); }
  ~Parquet_share() override { thr_lock_delete(&lock); }
};
```

`Handler_share`를 상속한 `Parquet_share`는 동일한 테이블에 대해 **하나만 생성**됩니다. 여러 커넥션이 같은 테이블을 열어도 `Parquet_share`는 공유됩니다.

`THR_LOCK`은 MariaDB의 테이블 레벨 락입니다. `thr_lock_init()`으로 초기화하고 `thr_lock_delete()`로 소멸시킵니다.

### ha_parquet — 핸들러 구현체

```cpp
class ha_parquet : public handler {
 public:
  ha_parquet(handlerton *hton, TABLE_SHARE *table_arg);

  const char *table_type() const override { return "READ_PARQUET"; }

  ulonglong table_flags() const override {
    return HA_NO_TRANSACTIONS | HA_BINLOG_STMT_CAPABLE;
  }

  // 쓰기 연산 — 모두 거부
  int write_row(const uchar *)                  override { return HA_ERR_WRONG_COMMAND; }
  int update_row(const uchar *, const uchar *)  override { return HA_ERR_WRONG_COMMAND; }
  int delete_row(const uchar *)                 override { return HA_ERR_WRONG_COMMAND; }

 private:
  THR_LOCK_DATA lock;
  Parquet_share *share{nullptr};

  std::shared_ptr<arrow::Table> arrow_table;  // 메모리에 올라온 Arrow 테이블
  int64_t current_row{0};                     // 풀스캔 커서
  int64_t total_rows{0};                      // 전체 행 수
};
```

- `table_flags()`: 이 엔진의 특성을 MySQL에 알립니다. `HA_NO_TRANSACTIONS`는 트랜잭션 미지원, `HA_BINLOG_STMT_CAPABLE`은 statement 기반 바이너리 로그 지원을 의미합니다.
- `write_row`, `update_row`, `delete_row`: 모두 `HA_ERR_WRONG_COMMAND`를 반환해 쓰기를 거부합니다.
- `arrow_table`: 파일 전체를 메모리에 올린 Arrow 테이블입니다.
- `current_row`: `rnd_next()` 호출마다 증가하는 스캔 커서입니다.

### MySQL 8.0 vs MariaDB 10.6 API 차이

같은 코드를 MySQL 8.0과 MariaDB 10.6에서 컴파일하려면 API 차이에 주의해야 합니다.

| 항목 | MySQL 8.0 | MariaDB 10.6 |
|------|-----------|-------------|
| `open()` 시그니처 | `open(..., const dd::Table*)` | `open(...)`  (dd 없음) |
| `create()` 시그니처 | `create(..., dd::Table*)` | `create(...)` (dd 없음) |
| 플러그인 등록 매크로 | `mysql_declare_plugin` | `maria_declare_plugin` |
| 헤더 경로 | `"sql/handler.h"` | `"handler.h"` |
| 디버그 매크로 | `DBUG_TRACE` | `DBUG_ENTER` / `DBUG_RETURN` |

이 책의 코드는 각각 별도 브랜치로 제공됩니다.

### Parquet 파일 열기

```cpp
#include <arrow/io/file.h>
#include <parquet/arrow/reader.h>
#include <arrow/table.h>

// 파일 경로로 Arrow ReadableFile 열기
auto result = arrow::io::ReadableFile::Open(parquet_path);
if (!result.ok()) { /* 오류 처리 */ }
std::shared_ptr<arrow::io::ReadableFile> infile = *result;

// Parquet FileReader 생성
std::unique_ptr<parquet::arrow::FileReader> reader;
auto status = parquet::arrow::OpenFile(infile,
                                       arrow::default_memory_pool(),
                                       &reader);
if (!status.ok()) { /* 오류 처리 */ }

// 전체 파일을 Arrow Table로 로드
std::shared_ptr<arrow::Table> table;
status = reader->ReadTable(&table);
```

Apache Arrow는 `Result<T>` 타입과 `Status` 타입으로 오류를 전달합니다. C++ 예외 대신 반환값으로 오류를 처리하는 스타일입니다.

## 5.4 핵심 메서드 구현

### create() — 사이드카 파일 생성

```cpp
int ha_parquet::create(const char *name, TABLE *form,
                       HA_CREATE_INFO *create_info)
{
    // CREATE TABLE ... COMMENT='경로' 에서 경로 추출
    const char *parquet_path = nullptr;
    if (create_info->comment.str && create_info->comment.length > 0) {
        parquet_path = create_info->comment.str;
    }

    // 사이드카 파일 경로: <MySQL 데이터 경로>.prq
    char prq_path[FN_REFLEN];
    snprintf(prq_path, sizeof(prq_path), "%s.prq", name);

    // parquet 파일 경로를 .prq 파일에 저장
    FILE *f = fopen(prq_path, "w");
    if (!f) return HA_ERR_GENERIC;
    if (parquet_path)
        fputs(parquet_path, f);
    else {
        // COMMENT 없으면 <테이블경로>.parquet 사용
        snprintf(...);
    }
    fclose(f);
    return 0;
}
```

`name`은 MySQL 데이터 디렉터리 내 테이블 경로입니다 (예: `/var/lib/mysql/mydb/sales`). 사이드카 파일은 이 경로에 `.prq` 확장자를 붙인 위치에 생성됩니다.

이 방식으로 MySQL 서버가 재시작되어도 parquet 파일 경로를 기억합니다.

### open() — Parquet 파일 로드

```cpp
int ha_parquet::open(const char *name, int mode, uint test_if_locked)
{
    // 1. 공유 상태 획득
    share = get_share();
    if (!share) return HA_ERR_OUT_OF_MEM;
    thr_lock_data_init(&share->lock, &lock, NULL);

    // 2. 사이드카 파일에서 parquet 경로 읽기
    char parquet_path[FN_REFLEN];
    if (!read_prq_meta(name, parquet_path))
        return HA_ERR_NO_SUCH_TABLE;

    // 3. Arrow로 parquet 파일 열기
    auto file_result = arrow::io::ReadableFile::Open(parquet_path);
    if (!file_result.ok()) return HA_ERR_NO_SUCH_TABLE;

    // 4. Parquet → Arrow Table 로드
    std::unique_ptr<parquet::arrow::FileReader> reader;
    parquet::arrow::OpenFile(*file_result,
                             arrow::default_memory_pool(), &reader);
    reader->ReadTable(&arrow_table);
    total_rows = arrow_table->num_rows();

    return 0;
}
```

`open()`은 `SELECT`가 시작될 때마다 호출됩니다. 매번 파일 전체를 메모리에 로드합니다. 대용량 파일에는 주의가 필요합니다.

### rnd_init() / rnd_next() / rnd_end() — 풀스캔

```cpp
int ha_parquet::rnd_init(bool scan) {
    current_row = 0;
    return 0;
}

int ha_parquet::rnd_next(uchar *buf) {
    if (current_row >= total_rows)
        return HA_ERR_END_OF_FILE;  // 스캔 종료

    int rc = fill_row(buf, current_row);
    if (rc == 0) current_row++;
    return rc;
}

int ha_parquet::rnd_end() {
    return 0;
}
```

풀스캔의 흐름은 단순합니다:
1. `rnd_init()`: 커서를 0으로 초기화
2. `rnd_next()`: 현재 행을 `buf`에 채우고 커서 증가. `HA_ERR_END_OF_FILE` 반환 시 스캔 종료
3. `rnd_end()`: 정리 (이 구현에서는 아무것도 하지 않음)

### fill_row() — Arrow → MySQL 타입 변환

`fill_row()`는 Arrow 테이블의 특정 행을 MySQL 행 버퍼(`buf`)에 복사하는 핵심 함수입니다.

```cpp
int ha_parquet::fill_row(uchar *buf, int64_t row_idx)
{
    // NULL 비트맵 초기화
    memset(buf, 0, table->s->null_bytes);

    for (uint i = 0; i < table->s->fields; i++) {
        Field *field = table->field[i];
        auto column = arrow_table->column(i);

        // ChunkedArray에서 스칼라 값 추출
        // Arrow Table의 컬럼은 ChunkedArray — 큰 파일을 청크 단위로 관리
        auto chunk_result = column->GetScalar(row_idx);
        if (!chunk_result.ok()) continue;
        auto scalar = *chunk_result;

        if (!scalar->is_valid) {
            // NULL 처리
            field->set_null();
            continue;
        }
        field->set_notnull();

        // 타입별 변환
        switch (column->type()->id()) {
        case arrow::Type::INT32:
            field->store(
                std::static_pointer_cast<arrow::Int32Scalar>(scalar)->value,
                false);
            break;
        case arrow::Type::DOUBLE:
            field->store(
                std::static_pointer_cast<arrow::DoubleScalar>(scalar)->value);
            break;
        case arrow::Type::STRING: {
            auto s = std::static_pointer_cast<arrow::StringScalar>(scalar);
            field->store(s->value->data(), s->value->size(),
                         &my_charset_utf8mb4_general_ci);
            break;
        }
        case arrow::Type::DATE32: {
            MYSQL_TIME mt{};
            days_to_mysql_date(
                std::static_pointer_cast<arrow::Date32Scalar>(scalar)->value,
                &mt);
            field->store_time(&mt);
            break;
        }
        // ... 그 외 타입들
        }
    }
    return 0;
}
```

#### Field::store() 메서드

`Field::store()`는 MySQL이 컬럼 타입에 맞게 값을 행 버퍼에 쓰는 메서드입니다.

| Arrow 타입 | Field::store() 호출 |
|-----------|---------------------|
| INT32/INT64 | `store(longlong, bool unsigned_flag)` |
| FLOAT/DOUBLE | `store(double)` |
| STRING | `store(const char*, ulong len, charset)` |
| DATE32 | `store_time(MYSQL_TIME*)` |
| BOOL | `store(0 or 1, false)` |

컬럼 순서(인덱스)로 매핑합니다. parquet 파일의 0번 컬럼이 MySQL DDL의 0번 컬럼에 대응됩니다. 이름 기반 매핑은 지원하지 않으므로, `CREATE TABLE`에서 컬럼 순서를 parquet 스키마와 일치시켜야 합니다.

### days_to_mysql_date() — 날짜 변환

Apache Arrow의 `DATE32` 타입은 Unix epoch(1970-01-01) 기준 일수(정수)입니다. MySQL의 `MYSQL_TIME` 구조체로 변환하려면 연/월/일을 계산해야 합니다.

Howard Hinnant의 Gregorian calendar 알고리즘을 사용합니다.

```cpp
static void ha_parquet::days_to_mysql_date(int32_t days, MYSQL_TIME *out)
{
    // 알고리즘: civil_from_days (Howard Hinnant)
    // https://howardhinnant.github.io/date_algorithms.html
    int64_t z = (int64_t)days + 719468;
    int64_t era = (z >= 0 ? z : z - 146096) / 146097;
    int64_t doe = z - era * 146097;
    int64_t yoe = (doe - doe/1460 + doe/36524 - doe/146096) / 365;
    int64_t y   = yoe + era * 400;
    int64_t doy = doe - (365*yoe + yoe/4 - yoe/100);
    int64_t mp  = (5*doy + 2) / 153;
    int64_t d   = doy - (153*mp + 2)/5 + 1;
    int64_t m   = mp < 10 ? mp+3 : mp-9;
    if (m <= 2) y += 1;

    memset(out, 0, sizeof(*out));
    out->year       = (uint)y;
    out->month      = (uint)m;
    out->day        = (uint)d;
    out->time_type  = MYSQL_TIMESTAMP_DATE;
}
```

## 5.5 빌드 시스템

### 왜 MySQL 소스가 필요한가

MySQL/MariaDB 바이너리 배포판에는 `mysql/plugin.h`, `handler.h`, `my_global.h` 등의 서버 내부 헤더가 포함되지 않습니다. 스토리지 엔진 개발을 위해서는 서버 소스코드에서 헤더를 가져와야 합니다.

중요한 제약이 있습니다. **서버 소스 버전이 설치된 MySQL/MariaDB 서버 버전과 정확히 일치해야 합니다.** 버전이 다르면 `INSTALL PLUGIN` 시 아래 오류가 발생합니다.

```
ERROR: API version for STORAGE ENGINE plugin is too different
```

### 선택적 빌드 방식

전체 MySQL 서버를 컴파일하는 것은 매우 오랜 시간이 걸립니다(수십 분~몇 시간). `build.sh`는 필요한 부분만 선택적으로 빌드합니다.

```
cmake configure   → 헤더 경로·플래그 생성만 (빌드 없음)
        ↓
make GenError     → mysqld_error.h 등 생성 헤더만 빌드
        ↓
make (build.make) → ha_parquet.cc.o 만 컴파일
        ↓
링크              → Arrow + Parquet 라이브러리만 링크
                  → MySQL 심볼은 INSTALL PLUGIN 시 mysqld가 제공
        ↓
ha_parquet.so / ha_read_parquet.so
```

`GenError` 타겟만 빌드하는 것이 핵심입니다. MySQL 서버 빌드 시스템이 `mysqld_error.h` 같은 생성 헤더를 만들어 주는데, 이것이 없으면 컴파일이 실패합니다.

### macOS 빌드

```bash
cd 05_ReadParquet/Mac

# 1단계: 의존성 설치
brew install apache-arrow openssl@3 lz4 cmake

# 2단계: MySQL 서버 소스 클론
git clone --depth=1 https://github.com/mysql/mysql-server.git
mysql_ver=$(/usr/local/mysql/bin/mysql_config --version)
git -C mysql-server fetch --tags
git -C mysql-server checkout mysql-${mysql_ver}

# 3단계: 빌드
./build.sh
```

성공 시 `ha_read_parquet.so` 생성.

### Linux 빌드

```bash
cd 05_ReadParquet/Linux

# 1단계: 의존성 설치
sudo apt install libarrow-dev libparquet-dev cmake g++ libssl-dev

# 2단계: MariaDB 소스 클론
```

성공 시 `ha_parquet.so` 생성.

## 5.6 사용 방법

### 플러그인 설치

```sql
-- macOS/MySQL
INSTALL PLUGIN read_parquet SONAME 'ha_read_parquet.so';

-- Linux/MariaDB
INSTALL PLUGIN read_parquet SONAME 'ha_parquet.so';

-- 설치 확인
SHOW PLUGINS;
```

### 테이블 생성

```sql
-- parquet 파일 경로를 COMMENT로 지정
CREATE TABLE sales (
    order_id   INT,
    product    VARCHAR(128),
    amount     DOUBLE,
    order_date DATE
) ENGINE=READ_PARQUET
  COMMENT='/data/warehouse/sales_2024.parquet';
```

컬럼 타입은 parquet 파일의 스키마와 호환되어야 합니다. 컬럼 **순서**가 일치해야 합니다 (이름 기반 매핑 아님).

### 테스트용 Parquet 파일 생성

Python으로 간단히 테스트 파일을 만들 수 있습니다.

```python
import pandas as pd

df = pd.DataFrame({
    'order_id':   [1, 2, 3],
    'product':    ['Apple', 'Banana', 'Cherry'],
    'amount':     [10.5, 20.0, 15.75],
    'order_date': pd.to_datetime(['2024-01-01', '2024-01-02', '2024-01-03'])
})
df.to_parquet('/tmp/sales.parquet', index=False)
```

### 쿼리

```sql
-- 전체 조회
SELECT * FROM sales;
-- +-----------+---------+--------+------------+
-- | order_id  | product | amount | order_date |
-- +-----------+---------+--------+------------+
-- |         1 | Apple   |   10.5 | 2024-01-01 |
-- |         2 | Banana  |   20.0 | 2024-01-02 |
-- |         3 | Cherry  | 15.75  | 2024-01-03 |
-- +-----------+---------+--------+------------+

-- 집계
SELECT product, SUM(amount) AS total
FROM sales
GROUP BY product;

-- 다른 엔진의 테이블과 조인
SELECT s.product, c.category
FROM sales s
JOIN product_catalog c ON s.product = c.name;
```

### 제거

```sql
DROP TABLE sales;
-- .prq 사이드카 파일도 자동 삭제됩니다

UNINSTALL PLUGIN read_parquet;
```

### 제약사항

| 항목 | 내용 |
|------|------|
| 쓰기 | 불가 — INSERT/UPDATE/DELETE 모두 `HA_ERR_WRONG_COMMAND` |
| 인덱스 | 미지원 — 항상 풀스캔 |
| 트랜잭션 | 미지원 |
| 컬럼 매핑 | 이름이 아닌 순서 기준 |
| 메모리 | 파일 전체를 메모리에 로드 |
| 버전 | MySQL/MariaDB 소스와 서버 버전이 일치해야 함 |

## 5장 정리

- 스토리지 엔진은 `handler` 추상 클래스를 상속받아 스토리지 계층 인터페이스를 구현한다.
- `Parquet_share`는 테이블 하나당 공유되는 락과 메타정보를 보관한다.
- `open()`에서 Arrow C++ 라이브러리로 Parquet 파일을 전체 메모리에 로드한다.
- `rnd_next()`에서 커서를 한 행씩 진행하며 `fill_row()`로 Arrow 컬럼 값을 MySQL Field에 저장한다.
- 사이드카 `.prq` 파일에 parquet 경로를 저장해 서버 재시작 후에도 경로를 유지한다.
- MySQL 소스 버전과 설치된 서버 버전이 반드시 일치해야 한다.
- `build.sh`로 전체 서버를 컴파일하지 않고 필요한 부분만 선택적으로 빌드한다.
