# 05-1. 스토리지 엔진 아키텍처

### MySQL의 스토리지 계층

MySQL은 SQL 파서, 옵티마이저, 스토리지 계층이 명확하게 분리된 구조입니다.

```
클라이언트 SQL
    ↓
파서 → 옵티마이저 → 실행 엔진
                        ↓
              ┌──────────────────────┐
              │  스토리지 엔진 추상층  │  ← handler 추상 클래스
              └──────────────────────┘
              ↙         ↓         ↘
           InnoDB    MyISAM    READ_PARQUET
```

쿼리 실행 엔진은 `handler` 추상 클래스의 인터페이스만 사용합니다. 어떤 스토리지 엔진인지는 신경 쓰지 않습니다. 따라서 `handler`를 상속받아 필요한 메서드를 구현하면 새로운 스토리지 엔진을 만들 수 있습니다.

이 장의 `READ_PARQUET`는 `.parquet` 파일을 MySQL 테이블처럼 쿼리할 수 있게 해주는 **읽기 전용** 스토리지 엔진입니다.

```sql
CREATE TABLE sales (
    order_id INT,
    product VARCHAR(128),
    amount DOUBLE
) ENGINE=READ_PARQUET
  COMMENT='/data/warehouse/sales_2024.parquet';

SELECT product, SUM(amount) FROM sales GROUP BY product;
```

### Apache Parquet 포맷

Apache Parquet는 컬럼 지향(columnar) 바이너리 파일 포맷입니다. 데이터 웨어하우스, 빅데이터 파이프라인에서 널리 사용됩니다.

- **컬럼 지향**: 같은 컬럼의 값들이 연속 저장 → 집계 쿼리에 효율적
- **스키마 내장**: 파일 자체에 컬럼명, 타입 정보가 포함
- **압축**: Snappy, Gzip 등 다양한 압축 지원

이 엔진은 Apache Arrow C++ 라이브러리(`libarrow`, `libparquet`)로 Parquet 파일을 파싱합니다.
