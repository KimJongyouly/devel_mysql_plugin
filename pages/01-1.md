# 01-1. UDF란 무엇인가

MySQL은 `COUNT()`, `SUM()`, `AVG()` 같은 내장 함수를 기본으로 제공합니다. 하지만 업무 현장에서는 이것만으로 해결되지 않는 요구사항이 종종 생깁니다.

예를 들어 이런 상황을 생각해봅시다.

- 사내 암호화 규칙에 따라 특정 컬럼 값을 변환해야 한다
- 외부 API를 호출해 반환값을 SQL 결과로 받고 싶다
- 중앙값, 최빈값 같은 통계 함수가 필요하다
- 머신러닝 모델의 추론 결과를 SQL 쿼리 안에서 쓰고 싶다

이런 경우 애플리케이션 코드에서 행마다 처리하면 수백만 건의 데이터를 네트워크로 전송해야 합니다. 반면 **UDF(User Defined Function)** 를 사용하면 그 처리를 MySQL 서버 안에서 직접 수행할 수 있습니다.

### UDF의 정의

UDF는 C 또는 C++로 작성한 공유 라이브러리(`.so` / `.dylib`)를 MySQL에 등록해, SQL 함수처럼 호출하는 확장 메커니즘입니다.

```sql
SELECT my_function(column) FROM table;
```

이 한 줄이 실제로는 여러분이 C++로 작성한 코드를 MySQL 프로세스 내부에서 직접 실행합니다.

### UDF의 종류

MySQL 플러그인 개발에서 다루는 함수 확장은 크게 세 가지로 나뉩니다.

| 종류 | 설명 | 예 |
|------|------|-----|
| **스칼라 UDF** | 행마다 하나의 값을 반환 | `my_encrypt(col)` |
| **집계 UDF** | 여러 행을 묶어 하나의 값을 반환, `GROUP BY` 지원 | `my_median(col)` |
| **스토리지 엔진** | 테이블 저장·조회 방식 자체를 구현 | `ENGINE=READ_PARQUET` |

이 책에서는 이 세 가지를 단계적으로 다룹니다. 먼저 가장 단순한 스칼라 UDF부터 시작합니다.

### UDF가 적합한 상황

UDF를 고려할 만한 상황은 다음과 같습니다.

- **성능**: 수백만 건 이상의 행을 변환·필터링할 때 애플리케이션보다 DB 내부 처리가 빠르다
- **재사용성**: 여러 애플리케이션·언어에서 동일한 로직을 DB 함수로 공유한다
- **통일성**: 암호화, 마스킹, 해시 등의 로직을 DB에서 한 곳에서 관리한다

반면 UDF가 적합하지 않은 상황도 있습니다. UDF 코드의 버그는 MySQL 서버 프로세스 전체를 크래시시킬 수 있습니다. 따라서 외부 API 호출, 파일 I/O, 복잡한 락 등 위험한 작업은 UDF보다 저장 프로시저나 애플리케이션 계층에서 처리하는 것이 안전합니다.
