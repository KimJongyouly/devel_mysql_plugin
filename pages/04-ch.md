# 제4장: 데몬 플러그인 — OS 리소스 모니터링

---

## 4.1 데몬 플러그인이란

### UDF와 다른 플러그인 세계

앞의 세 장에서는 SQL 함수로 호출하는 UDF를 만들었습니다. UDF는 쿼리가 들어올 때만 실행됩니다. **데몬 플러그인(Daemon Plugin)** 은 다릅니다. MySQL 서버가 시작될 때 함께 시작되어, 서버가 종료될 때까지 백그라운드에서 독립적으로 실행됩니다.

데몬 플러그인의 대표적인 활용 사례:

- **리소스 모니터링**: CPU, 메모리, 디스크 I/O를 주기적으로 로그에 기록
- **백그라운드 정리 작업**: 만료된 세션, 임시 파일을 주기적으로 삭제
- **외부 시스템 연동**: 메트릭을 외부 모니터링 시스템(Prometheus 등)으로 주기적 전송
- **감사 로그**: 쿼리 통계를 비동기적으로 기록

### MySQL 플러그인 종류

MySQL/MariaDB는 UDF 외에도 여러 플러그인 타입을 지원합니다.

| 플러그인 타입 | 상수 | 설명 |
|------------|------|------|
| UDF | (별도 등록) | SQL 함수 확장 |
| Daemon | `MYSQL_DAEMON_PLUGIN` | 백그라운드 작업 |
| Storage Engine | `MYSQL_STORAGE_ENGINE_PLUGIN` | 스토리지 계층 |
| Full-text Parser | `MYSQL_FTPARSER_PLUGIN` | 전문 검색 파서 |
| Audit | `MYSQL_AUDIT_PLUGIN` | 감사 로그 |
| Authentication | `MYSQL_AUTHENTICATION_PLUGIN` | 인증 확장 |

이 장에서는 `MYSQL_DAEMON_PLUGIN` 타입을 사용합니다.

### UDF와의 구조적 차이

| 항목 | UDF | 데몬 플러그인 |
|------|-----|------------|
| 등록 방법 | `CREATE FUNCTION` | `INSTALL PLUGIN` |
| 실행 트리거 | SQL 호출 시 | 플러그인 설치 즉시 |
| 실행 공간 | SQL 실행 컨텍스트 | 별도 백그라운드 스레드 |
| 헤더 파일 | `mysql.h` | `mysql.h` + `mysql/plugin.h` |
| 플러그인 등록 | (없음, `_init/_deinit` 직접 내보내기) | `mysql_declare_plugin` 매크로 |

## 4.2 플러그인 등록 구조

### plugin.h의 플러그인 descriptor

데몬 플러그인은 `mysql/plugin.h`에 정의된 매크로와 구조체를 사용합니다.

MySQL 8.0에서는 `mysql_declare_plugin`, MariaDB에서는 `maria_declare_plugin` 매크로를 사용합니다.

**MariaDB 버전 (Linux):**

```cpp
struct st_mysql_daemon os_logger_info = { MYSQL_DAEMON_INTERFACE_VERSION };

maria_declare_plugin(os_logger)
{
    MYSQL_DAEMON_PLUGIN,    // 플러그인 타입
    &os_logger_info,        // 타입별 인터페이스 구조체
    "os_logger",            // 플러그인 이름 (INSTALL PLUGIN에서 사용)
    "youly",                // 작성자
    "OS Resource Logger",   // 설명
    PLUGIN_LICENSE_GPL,     // 라이선스
    os_logger_init,         // 초기화 함수 (INSTALL PLUGIN 시 호출)
    os_logger_deinit,       // 종료 함수 (UNINSTALL PLUGIN 시 호출)
    0x0100,                 // 버전 (1.0)
    NULL,                   // SHOW STATUS 변수 (없으면 NULL)
    NULL,                   // 시스템 변수 (없으면 NULL)
    "1.0",                  // 버전 문자열 (MariaDB 전용)
    MariaDB_PLUGIN_MATURITY_STABLE  // 안정성 등급 (MariaDB 전용)
}
maria_declare_plugin_end;
```

`MYSQL_DAEMON_INTERFACE_VERSION`은 현재 MySQL/MariaDB API 버전을 나타내는 상수입니다. 빌드 시 사용한 헤더의 버전과 설치된 서버의 버전이 맞지 않으면 `INSTALL PLUGIN` 시 오류가 발생합니다.

## 4.3 백그라운드 스레드 구현

### Linux 버전 전체 코드

Linux의 `os_logger`는 `sysinfo()` 시스템 콜로 CPU 부하와 메모리 사용량을 측정합니다.

```cpp
#include <mysql.h>
#include <mysql/plugin.h>
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <pthread.h>
#include <sys/sysinfo.h>

static pthread_t logger_thread;
static volatile int stop_logger = 0;
static int thread_running = 0;

// 백그라운드 워커 스레드
static void *os_logger_worker(void *p) {
    struct sysinfo info;
    while (!stop_logger) {
        if (sysinfo(&info) == 0) {
            double load = info.loads[0] / 65536.0;
            unsigned long total_ram = (info.totalram * info.mem_unit) / 1024 / 1024;
            unsigned long free_ram  = (info.freeram  * info.mem_unit) / 1024 / 1024;
            fprintf(stderr, "[os_logger] CPU Load: %.2f, RAM: %lu/%lu MB\n",
                    load, free_ram, total_ram);
        }
        for (int i = 0; i < 5 && !stop_logger; i++) sleep(1);
    }
    return NULL;
}

// INSTALL PLUGIN 시 호출
static int os_logger_init(void *p) {
    stop_logger    = 0;
    thread_running = 0;

    int ret = pthread_create(&logger_thread, NULL, os_logger_worker, NULL);
    if (ret != 0) {
        fprintf(stderr, "[os_logger] pthread_create failed: %d\n", ret);
        return 1;
    }
    thread_running = 1;
    return 0;
}

// UNINSTALL PLUGIN 시 호출
static int os_logger_deinit(void *p) {
    stop_logger = 1;
    if (thread_running) {
        pthread_join(logger_thread, NULL);
        thread_running = 0;
    }
    return 0;
}

extern "C" {
    struct st_mysql_daemon os_logger_info = { MYSQL_DAEMON_INTERFACE_VERSION };

    maria_declare_plugin(os_logger)
    { ... }
    maria_declare_plugin_end;
}
```

### 코드 상세 분석

#### 전역 변수와 `volatile`

```cpp
static pthread_t logger_thread;
static volatile int stop_logger = 0;
static int thread_running = 0;
```

- `logger_thread`: 생성된 스레드의 ID를 저장합니다. `pthread_join()`에서 사용합니다.
- `stop_logger`: 메인 스레드(deinit)가 워커 스레드에게 종료 신호를 보내는 플래그입니다. `volatile`을 붙여 컴파일러가 이 변수를 레지스터에 캐시하지 않도록 합니다.
- `thread_running`: 스레드가 실제로 생성됐는지 추적합니다. `pthread_join()`을 이중 호출하는 것을 방지합니다.

> **`volatile`의 역할**: 컴파일러는 최적화를 위해 반복 루프에서 변하지 않는 변수를 레지스터에 캐시합니다. `stop_logger`는 다른 스레드에서 변경되므로, `volatile` 없이는 워커 스레드가 갱신된 값을 읽지 못할 수 있습니다.

#### 워커 스레드 종료 전략

```cpp
for (int i = 0; i < 5 && !stop_logger; i++) sleep(1);
```

단순히 `sleep(5)`를 쓰면 `deinit()`이 호출됐을 때 최대 5초를 기다려야 합니다. 위 코드는 1초씩 5번 나눠 자면서 매 초마다 `stop_logger` 플래그를 확인합니다. `deinit()`이 호출되면 최대 1초 안에 반응합니다.

#### 안전한 종료: `pthread_join()`

```cpp
static int os_logger_deinit(void *p) {
    stop_logger = 1;           // 종료 신호
    if (thread_running) {
        pthread_join(logger_thread, NULL);  // 완전히 끝날 때까지 대기
        thread_running = 0;
    }
    return 0;
}
```

`stop_logger = 1`은 워커 스레드에게 종료 신호를 보낼 뿐입니다. `pthread_join()`이 워커 스레드가 완전히 종료될 때까지 `deinit()`을 블로킹합니다.

이것이 중요한 이유가 있습니다. `UNINSTALL PLUGIN` 이후 플러그인 코드는 메모리에서 언로드됩니다. 만약 워커 스레드가 아직 실행 중인 상태에서 코드가 언로드되면 **세그폴트(segfault)** 가 발생합니다. `pthread_join()`으로 스레드가 완전히 종료됨을 보장해야 합니다.

## 4.3 OS 메트릭 수집

### Linux: sysinfo() 시스템 콜

Linux에서는 `<sys/sysinfo.h>`의 `sysinfo()` 함수로 시스템 자원 정보를 가져옵니다.

```c
struct sysinfo {
    long uptime;            // 부팅 후 경과 시간 (초)
    unsigned long loads[3]; // 1분, 5분, 15분 평균 부하 (65536 기준)
    unsigned long totalram; // 전체 RAM (mem_unit 단위)
    unsigned long freeram;  // 여유 RAM (mem_unit 단위)
    unsigned long sharedram;
    unsigned long bufferram;
    unsigned long totalswap;
    unsigned long freeswap;
    unsigned short procs;   // 실행 중인 프로세스 수
    unsigned long mem_unit; // 메모리 단위 (바이트, 보통 1)
};
```

CPU 부하 계산:

```cpp
double load = info.loads[0] / 65536.0;
```

`loads[0]`은 1분 평균 부하를 65536으로 스케일링한 고정소수점 값입니다. 65536으로 나누면 실제 load average를 얻습니다.

메모리 계산:

```cpp
unsigned long total_ram = (info.totalram * info.mem_unit) / 1024 / 1024;
unsigned long free_ram  = (info.freeram  * info.mem_unit) / 1024 / 1024;
```

`totalram`에 `mem_unit`을 곱하면 바이트가 됩니다. 1024로 두 번 나누면 MB 단위입니다.

### macOS: Mach API와 IOKit

macOS 버전(`my_resource`)은 훨씬 더 많은 메트릭을 수집합니다.

**CPU 사용률** — Mach API `host_processor_info()`:

```cpp
// CPU 정보 수집 (user + sys 시간 비율로 계산)
processor_info_array_t cpu_info;
mach_msg_type_number_t num_cpus_info;
natural_t num_cpus;

host_processor_info(mach_host_self(), PROCESSOR_CPU_LOAD_INFO,
                    &num_cpus, &cpu_info, &num_cpus_info);
```

1초 간격으로 두 번 측정해 그 차이로 CPU 사용률을 계산합니다 (idle 시간 제외).

**메모리 사용량** — `vm_statistics64`:

```cpp
vm_statistics64_data_t vmstat;
mach_msg_type_number_t count = HOST_VM_INFO64_COUNT;
host_statistics64(mach_host_self(), HOST_VM_INFO64,
                  (host_info64_t)&vmstat, &count);

// 사용 중인 페이지 = active + wired
uint64_t used_pages = vmstat.active_count + vmstat.wire_count;
uint64_t used_mb = (used_pages * page_size) / 1024 / 1024;
```

**디스크 I/O** — IOKit `IOBlockStorageDriver`:

macOS의 IOKit 프레임워크를 통해 블록 스토리지 드라이버에서 누적 읽기/쓰기 바이트를 가져오고, 이전 측정값과의 차이로 초당 속도를 계산합니다.

## 4.4 MySQL STATUS 변수 노출 (macOS 버전)

macOS의 `my_resource` 플러그인은 수집한 메트릭을 `SHOW STATUS`로 조회할 수 있도록 STATUS 변수로도 노출합니다.

### STATUS 변수 등록

```cpp
// 전역 상태 변수
static double   g_cpu_pct      = 0.0;
static long long g_mem_used_mb = 0;
static long long g_mem_total_mb = 0;
// ...

// STATUS 변수 정의 배열
static SHOW_VAR my_resource_status_vars[] = {
    {"my_resource_cpu_pct",
     (char*)&g_cpu_pct, SHOW_DOUBLE, SHOW_SCOPE_GLOBAL},
    {"my_resource_mem_used_mb",
     (char*)&g_mem_used_mb, SHOW_LONGLONG, SHOW_SCOPE_GLOBAL},
    {"my_resource_mem_total_mb",
     (char*)&g_mem_total_mb, SHOW_LONGLONG, SHOW_SCOPE_GLOBAL},
    // ...
    {NullS, NullS, SHOW_LONG, SHOW_SCOPE_GLOBAL}  // 종료 마커
};
```

이 배열을 플러그인 descriptor에 등록합니다.

```cpp
mysql_declare_plugin(my_resource)
{
    MYSQL_DAEMON_PLUGIN,
    &my_resource_info,
    "my_resource",
    "youly",
    "MySQL OS Resource Monitor",
    PLUGIN_LICENSE_GPL,
    my_resource_init,
    nullptr,              // check_uninstall (MySQL 8.0 추가 필드)
    my_resource_deinit,
    0x0101,
    my_resource_status_vars,  // STATUS 변수 등록
    my_resource_sys_vars,     // 시스템 변수 등록
    nullptr,
    0
}
mysql_declare_plugin_end;
```

### 시스템 변수 (SET GLOBAL)

모니터링 주기를 런타임에 변경할 수 있도록 시스템 변수도 등록합니다.

```cpp
static int g_interval = 5;  // 기본값: 5초

static MYSQL_SYSVAR_INT(
    interval,          // 변수 이름 → my_resource_interval
    g_interval,
    PLUGIN_VAR_RQCMDARG,
    "Monitoring interval in seconds",
    NULL, NULL,
    5, 1, 3600, 0      // 기본값, 최소, 최대, 블록 크기
);
```

등록 후:

```sql
-- 현재 주기 확인
SHOW STATUS LIKE 'my_resource%';

-- 주기를 10초로 변경
SET GLOBAL my_resource_interval = 10;
```

## 4.5 빌드와 설치

### Linux 빌드

```bash
cd 04_OS_logger/Linux
make
make install
```

Makefile은 MariaDB 11.4.3의 헤더를 직접 지정합니다:

```makefile
MARIADB_BASE = /home/mariadb/mariadb_11.4.3
CXXFLAGS = -fPIC -Wall -DMYSQL_DYNAMIC_PLUGIN \
           -I$(MARIADB_BASE)/include/server/mysql \
           -I$(MARIADB_BASE)/include/mysql/server
LDFLAGS  = -shared -lpthread
```

### macOS 빌드

macOS 버전은 MySQL 서버 소스 헤더(`mysql/plugin.h`)가 필요합니다. MySQL 바이너리 배포판에는 이 헤더가 포함되지 않습니다.

```bash
cd 04_OS_logger/Mac
make setup   # GitHub에서 MySQL include/ 디렉터리만 sparse-checkout
make         # 컴파일
sudo make install
```

### 플러그인 설치

```sql
-- Linux/MariaDB
INSTALL PLUGIN os_logger SONAME 'os_logger.so';

-- macOS/MySQL
INSTALL PLUGIN my_resource SONAME 'my_resource.so';

-- 설치 확인
SELECT PLUGIN_NAME, PLUGIN_STATUS
FROM information_schema.PLUGINS
WHERE PLUGIN_NAME IN ('os_logger', 'my_resource');
```

### 동작 확인

Linux:
```bash
tail -f /home/mariadb/mariadb_11.4.3/data/*.err
# [os_logger] CPU Load: 0.23, RAM: 3412/7982 MB
```

macOS:
```bash
sudo tail -f /usr/local/mysql/data/*.err
# [Note] my_resource: CPU=4.2% | MEM=7821/16384MB | DISK_R=12KB/s ...
```

macOS 버전은 STATUS 변수로도 확인 가능합니다:

```sql
SHOW STATUS LIKE 'my_resource%';
-- +------------------------------+----------+
-- | Variable_name                | Value    |
-- +------------------------------+----------+
-- | my_resource_cpu_pct          | 4.200000 |
-- | my_resource_mem_used_mb      | 7821     |
-- | my_resource_mem_total_mb     | 16384    |
-- +------------------------------+----------+
```

### 플러그인 제거

```sql
UNINSTALL PLUGIN os_logger;
-- deinit()이 호출되어 백그라운드 스레드가 안전하게 종료됩니다
```

### 서버 시작 시 자동 로드

`my.cnf`에 추가하면 MySQL/MariaDB 시작 시 자동으로 로드됩니다:

```ini
[mysqld]
plugin-load-add = os_logger=os_logger.so
```

## 4장 정리

- 데몬 플러그인은 `INSTALL PLUGIN`으로 로드되어 서버가 종료될 때까지 백그라운드에서 실행된다.
- `mysql_declare_plugin` / `maria_declare_plugin` 매크로로 플러그인을 등록하고 `_init`, `_deinit` 함수를 연결한다.
- `pthread_create()`로 백그라운드 스레드를 시작하고, `volatile int` 플래그로 종료 신호를 보낸다.
- `pthread_join()`으로 `_deinit()`에서 스레드가 완전히 종료됨을 보장해야 플러그인 코드가 안전하게 언로드된다.
- Linux에서는 `sysinfo()`, macOS에서는 Mach API와 IOKit으로 OS 메트릭을 수집한다.
- STATUS 변수와 시스템 변수를 등록해 `SHOW STATUS`, `SET GLOBAL`로 플러그인을 제어할 수 있다.

다음 장에서는 MySQL의 스토리지 계층을 직접 구현하는 커스텀 스토리지 엔진을 만듭니다.
